🧮 Fundamentos Matemáticos do ELOIDA OS

1. O Problema da Colisão Periódica

Em schedulers cooperativos tradicionais baseados em tick counters, é comum definir intervalos como potências de 2 (100, 200, 500 ms). O problema é que esses valores compartilham fatores comuns, o que leva a colisões periódicas:

T1 executa a cada 100 ticks
T2 executa a cada 200 ticks
T3 executa a cada 400 ticks

MDC(100, 200, 400) = 100
→ A cada 400 ticks, AS TRÊS tarefas executam juntas!
→ Pico de carga → latência → starvation

Esse fenômeno é conhecido como ressonância harmônica em sistemas de tempo real, e é uma das principais causas de jitter em firmwares mal projetados.

2. A Solução: Coprimalidade dos Primos

Dois números são coprimos quando seu máximo divisor comum é 1. Todo par de números primos distintos é, por definição, coprimo:

Se p ≠ q são primos, então MDC(p, q) = 1

Exemplo: MDC(17, 23) = 1
→ O ciclo conjunto só se repete a cada 17 × 23 = 391 iterações
→ As execuções ficam "espalhadas" uniformemente no tempo

2.1 Teorema Fundamental

Dadas n tarefas com períodos p₁, p₂, ..., pₙ onde todos são primos distintos, o LCM (mínimo múltiplo comum) dos períodos é:

LCM(p₁, p₂, ..., pₙ) = p₁ × p₂ × ... × pₙ

Isso significa que a coincidência total de todas as tarefas ocorre apenas após um número astronomicamente grande de iterações, garantindo distribuição suave da carga.

2.2 Exemplo Prático no ELOIDA OS

TarefaPeríodo (primo)Executa a cada...
task0 (sensor)22 ciclos
task1 (filtro)33 ciclos
task2 (atuador)55 ciclos
LCM(2, 3, 5) = 30
Em 30 iterações: task0 executa 15×, task1 executa 10×, task2 executa 6× — mas nunca todas juntas em intervalos curtos.

3. Por que 255 Primos?

O kernel suporta até MAX_PROCESSES = 255, o que permite indexar cada processo com um u8 (1 byte). A tabela PRIME_NUMS[255] contém os primeiros 255 números primos, culminando em 1627 (o 255º primo).

255º primo = 1627
Maior período possível = 1627 iterações
Menor período = 2 iterações
Razão máxima ≈ 813×

Essa faixa oferece granularidade suficiente para diferenciar tarefas de alta frequência (leituras de sensor) de tarefas esporádicas (logs, watchdog).

3.1 Fallback para tarefas 3 a 254

Para não "queimar" primos valiosos em tarefas que não exigem precisão matemática, o kernel usa a fórmula:

período(i) = 1000 + i, para i ≥ 3

Isso mantém as 3 primeiras tarefas (críticas) com períodos primos curtos, e as demais com períodos longos e espaçados — uma otimização heurística elegante.

4. Economia de Recursos: A Filosofia

O ELOIDA OS aplica três princípios de design minimalista:

4.1 Princípio da Estática Total

Toda estrutura de dados é alocada em tempo de compilação. Não há malloc, free, nem fragmentação de heap.

4.2 Princípio da Ausência de Mutex

Como o scheduler é cooperativo e de thread única (loop sequencial), não existem condições de corrida reais. O IPC por SharedData é inerentemente seguro.

4.3 Princípio do Despacho O(1)

A TASK_TABLE[12] é um array de ponteiros para função indexado diretamente pelo ID do processo. Nenhuma busca, nenhuma fila, nenhuma prioridade dinâmica.

Resultado: o kernel completo cabe em menos de 2 KB de RAM e 3 KB de flash, sendo executável em MCUs de entrada como o CH32V003 (2 KB RAM).
