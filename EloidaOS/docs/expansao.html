📈 Como Expandir o ELOIDA OS

1. Filosofia de Expansão

O ELOIDA OS foi projetado para crescer sem quebrar sua essência minimalista. Toda expansão deve respeitar três regras:

📏 Regras de Ouro

  1. Nenhuma alocação dinâmica — tudo estático em compilação
  2. Nenhuma premptividade — tarefas sempre cooperativas
  3. Nenhuma abstração desnecessária — se dá pra fazer em C direto, faça

2. Aumentando o número de tarefas nativas (12 → N)

A TASK_TABLE tem 12 slots. Para expandir:

#define TASK_COUNT 12
#define TASK_COUNT 32

// Adicione novas funções
void task12(void) { /* ... */ }
void task13(void) { /* ... */ }
// ... até task31

const TaskFunc TASK_TABLE[TASK_COUNT] = {
    task0,  task1,  task2,  task3,  task4,  task5,
    task6,  task7,  task8,  task9,  task10, task11,
    task12, task13, task14, task15, task16, task17,
    task18, task19, task20, task21, task22, task23,
    task24, task25, task26, task27, task28, task29,
    task30, task31
};

// Ajuste o fallback no run()
void run(u8 any_process) {
    if (any_process < MAX_PROCESSES && PROCESS[any_process] == 1) {
        if (any_process < 12)
        if (any_process < TASK_COUNT)
            TASK_TABLE[any_process]();
    }
}

3. Adicionando prioridades (extensão opcional)

O kernel original não tem prioridades — todas as tarefas são iguais. Para adicionar um sistema simples de prioridades por peso:

typedef struct {
    TaskFunc func;
    u8 priority;    // 1 = baixa, 10 = alta
    u8 enabled;     // 0 = desabilitada
} TaskEntry;

TaskEntry TASK_TABLE[12] = {
    {task0, 10, 1},  // sensor: alta prioridade
    {task1,  8, 1},  // filtro
    {task2,  9, 1},  // atuador
    {task3,  3, 1},  // status: baixa
    // ...
};

void run(u8 i) {
    if (PROCESS[i] && TASK_TABLE[i].enabled) {
        TASK_TABLE[i].func();
    }
}

💡 Ideia: escalonamento adaptativo

Tarefas que demoram muito para executar podem ter seu período primo aumentado dinamicamente, enquanto tarefas rápidas recebem primos menores. O resultado é um auto-tuning baseado em carga real.

4. IPC Avançado: Mensagens Tipadas

O SharedData é ótimo para fluxos lineares, mas para comunicação genérica entre tarefas, use um barramento de mensagens:

#define MSG_QUEUE_SIZE 16

typedef struct {
    u8 sender;
    u8 receiver;
    u8 type;
    u8 payload[8];
} Message;

typedef struct {
    Message buf[MSG_QUEUE_SIZE];
    u8 head;
    u8 tail;
    u8 count;
} MessageQueue;

MessageQueue MSG_BUS = {0};

void msg_send(u8 from, u8 to, u8 type, u8 *data, u8 len) {
    if (MSG_BUS.count >= MSG_QUEUE_SIZE) return;
    Message *m = &MSG_BUS.buf[MSG_BUS.head];
    m->sender = from;
    m->receiver = to;
    m->type = type;
    for (u8 i = 0; i < len && i < 8; i++) m->payload[i] = data[i];
    MSG_BUS.head = (MSG_BUS.head + 1) % MSG_QUEUE_SIZE;
    MSG_BUS.count++;
}

bool msg_receive(u8 who, Message *out) {
    for (u8 i = 0; i < MSG_BUS.count; i++) {
        u8 idx = (MSG_BUS.tail + i) % MSG_QUEUE_SIZE;
        if (MSG_BUS.buf[idx].receiver == who) {
            *out = MSG_BUS.buf[idx];
            // remove compactando
            MSG_BUS.tail = (MSG_BUS.tail + 1) % MSG_QUEUE_SIZE;
            MSG_BUS.count--;
            return true;
        }
    }
    return false;
}

5. Adicionando Drivers Periféricos

🎛️ Driver de ADC (task0 real)

void ADC_Init(void) {
    RCC->APB2PCENR |= RCC_APB2Periph_ADC1;
    ADC1->CTLR2 |= ADC_ADON;
}

void task0(void) {
    ADC1->RSQR3 = 0;          // canal 0
    ADC1->CTLR2 |= ADC_SWSTART;
    while (!(ADC1->STATR & ADC_EOC));
    SHARED.sensor_raw = ADC1->RDATAR;
}

⚡ Driver de PWM (task2 real)

void task2(void) {
    u16 duty = SHARED.actuator_cmd ? 2048 : 0;
    TIM2->CH1CVR = duty;  // 50% ou 0%
}

📡 Driver I2C (para sensores externos)

void task0(void) {
    u8 buf[2];
    I2C_ReadReg(0x68, 0x3B, buf, 2);  // ex: MPU6050
    SHARED.sensor_raw = (buf[0] << 8) | buf[1];
}

8. Como Contribuir com o Projeto

  1. Fork o repositório no GitHub
  2. Crie uma branch: git checkout -b feature/nova-task
  3. Implemente seguindo as 3 Regras de Ouro (seção 1)
  4. Teste em hardware real (simuladores não capturam timing)
  5. Documente em docs/ se for uma feature nova
  6. Abra um Pull Request descrevendo o impacto em RAM/Flash
