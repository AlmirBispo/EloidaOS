2. Estruturas de Dados do Kernel

🗂️ PROCESS[255]

Array de flags u8. Quando PROCESS[i] == 1, a tarefa i está pronta para executar no ciclo atual.

Tamanho: 255 bytes em RAM

🔢 COUNTS[255]

Contadores u32 incrementados a cada iteração do main loop. Quando atingem o limite, disparam a tarefa.

Tamanho: 1020 bytes em RAM

🧮 PRIME_NUMS[255]

Os primeiros 255 números primos, de 2 até 1627. Usados como limites para os 3 primeiros processos críticos.

Tamanho: 1020 bytes em flash (const)

📡 SHARED (SharedData)

Estrutura global de IPC com campos tipados. Preenchida sequencialmente pelas tarefas 0, 1 e 2.

Tamanho: ~44 bytes em RAM

5. API do Kernel

void process_create(u8 pointer);              // Inicializa um processo
void process(u32 Limit_count, u8 pointer);    // Incrementa e verifica
void run(u8 any_process);                     // Despacha via tabela
void destroy(u8 any_process);                 // Limpa a flag

Essas 4 funções formam a ABI mínima do ELOIDA OS — tudo o que o usuário precisa saber para usar o kernel.

6. Por que não preemptivo?

A escolha por um scheduler cooperativo (não preemptivo) é intencional e fundamentada:

  • ✅ Elimina necessidade de salvamento de contexto (economiza ciclos)
  • ✅ Elimina condições de corrida no IPC (economia de mutexes)
  • ✅ Compatível com qualquer toolchain RISC-V sem assembly customizado
  • ✅ Previsibilidade total: cada tarefa sabe exatamente quando vai rodar
  • ⚠️ Limitação: tarefas não podem bloquear (usar polling ou flags)
