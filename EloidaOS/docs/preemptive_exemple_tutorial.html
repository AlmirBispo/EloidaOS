ELOIDA OS

Guia Avançado: Implementando Preempção Real baseada em Interrupções de Hardware

O Conceito

Por padrão, o ELOIDA OS opera de forma cooperativa estrita, o que garante consumo zero de overhead na CPU RISC-V e previsibilidade matemática através de números primos. No entanto, se o seu projeto possui tarefas longas que não podem reter a CPU durante um evento crítico, você pode implementar a preempção forçada por hardware.

A estratégia consiste em usar os controladores de interrupção física do chip (como o PFIC/NVIC do CH32V307) para capturar um evento de pino externo e, de dentro da Rotina de Interrupção (ISR), zerar a flag de execução da tarefa corrente.

Como funciona o fluxo de interrupção:

  1. Uma tarefa longa (ex: Processo 11) está executando seu laço interno na CPU.
  2. Um sensor externo ou botão crítico gera uma transição de sinal em uma porta física (PA0 ou PA1).
  3. O hardware do RISC-V congela a CPU instantaneamente e desvia o fluxo para a função de interrupção (EXTI_IRQHandler).
  4. Dentro da interrupção, o código faz: PROCESS[11] = 0;
  5. A interrupção termina e a CPU volta para a tarefa longa.
  6. No próximo ponto de checagem (Checkpoint) dentro da tarefa longa, ela percebe que sua flag foi zerada e encerra imediatamente, liberando o laço do OS.

Requisitos de Implementação

1. Volatilidade Obrigatória

O array de processos do kernel deve ser declarado com o modificador volatile u8 PROCESS[MAX_PROCESSES]. Isso impede que o compilador GCC guarde a flag em registradores internos e a force a ler direto da RAM a cada ciclo.

2. Pontos de Checagem (Checkpoints)

Tarefas longas devem ser escritas de forma amigável à interrupção. O desenvolvedor deve espalhar checagens rápidas dentro de laços pesados para que a resposta da desistência seja o mais rápida possível.

Exemplo Prático em C

Abaixo está a estrutura padrão que o usuário deve adotar no código para integrar o ELOIDA OS com as interrupções de hardware: 

// 1. Definição da Macro de Checkpoint para o usuário
#define ELOIDA_CHECKPOINT(id) if (PROCESS[id] == 0) return;

// 2. Interrupção de hardware monitorando uma porta física
void EXTI0_IRQHandler(void) __attribute__((interrupt("WCH-Interrupt-fast")));
void EXTI0_IRQHandler(void) {
    if (EXTI_GetITStatus(EXTI_Line0) != RESET) {
        
        PROCESS[11] = 0; // Força a destruição imediata do fluxo do Processo 11
        
        // Trata o evento crítico da porta aqui com prioridade máxima...
        SHARED.sensor_port_A++; 
        
        EXTI_ClearITPendingBit(EXTI_Line0); // Limpa o registrador de interrupção
    }
}

// 3. Escrita da tarefa longa utilizando os checkpoints estratégicos
void task11(void) {
    for (u32 i = 0; i < 10000; i++) {
        
        ELOIDA_CHECKPOINT(11); // Se a porta disparou a ISR, sai da função na hora!
        
        // Lógica pesada de processamento do usuário aqui...
        ProcessarDadosComplexos();
        
        ELOIDA_CHECKPOINT(11); // Segundo ponto de checagem antes de reprocessar
    }
}

Vantagens dessa Abordagem

  • Simplicidade de Código: O kernel do ELOIDA OS permanece idêntico, leve e focado em recursos mínimos, sem necessidade de rotinas complexas de troca de contexto de pilha assembly.
  • Latência Mínima: O evento físico da porta é respondido de forma imediata dentro da ISR, enquanto o descarte da tarefa longa ocorre logo em seguida no nível do software de forma limpa.
  • Robustez Industrial: Evita o efeito cascata onde uma falha de comunicação de barramento ou um loop de processamento longo em uma tarefa paralise as leituras de segurança das portas principais.
