ELOIDA OS

Arquitetura de Escalonamento por Primos e Controle de Preempção por Hardware

O Conceito: Escalonamento vs Temporização

No ELOIDA OS, a contagem de ciclos baseada em números primos não tem o objetivo de funcionar como temporizador (timer). O foco é puramente escalar as tarefas e evitar a sobreposição de execução no mesmo ciclo do laço principal.

Como os números primos possuem propriedades matemáticas únicas de não-divisibilidade entre si, os momentos em que as tarefas atingem seus limites de contagem são dispersos de forma natural ao longo do tempo de processamento. Isso reduz drasticamente a chance de múltiplas tarefas pesadas tentarem rodar na mesma iteração, otimizando o uso do hardware sem gerar o overhead de um sistema operacional convencional.

Interrompendo uma Tarefa Longa via Hardware

Embora o escalonamento matemático distribua as tarefas perfeitamente para evitar gargalos, o usuário pode se deparar com cenários onde uma tarefa longa de prioridade escalar menor (como processar logs ou enviar pacotes extensos) precise ser cancelada imediatamente por um evento externo.

A arquitetura enxuta do ELOIDA OS permite criar uma preempção baseada em interrupção de forma muito simples: basta configurar uma interrupção física (ISR) para monitorar as portas de hardware e zerar a flag PROCESS[id] = 0 do processo corrente, limpando o gatilho de execução.

Fluxo da Interrupção Reativa:

  1. Uma tarefa longa do usuário está sendo executada.
  2. Um evento físico em uma ou mais portas (sensores) dispara uma interrupção de hardware no RISC-V.
  3. O hardware desvia a execução para a ISR correspondente, que zera a flag daquela tarefa longa (PROCESS[id] = 0;).
  4. Ao retornar para a tarefa, os pontos de checagem (Checkpoints) inseridos pelo usuário detectam a flag zerada e realizam um return imediato, devolvendo a CPU para o laço do OS de forma limpa.

Implementação Prática em C (CH32V307)

Abaixo está o modelo de código que demonstra como o usuário pode proteger o sistema contra retenção de CPU, monitorando duas portas por interrupção: 

// Definição da Macro de Checkpoint para uso em loops longos
#define ELOIDA_CHECKPOINT(id) if (PROCESS[id] == 0) return;

#define LONG_TASK_ID 11

// ISR da Porta 1 (Sensor Crítico A)
void EXTI0_IRQHandler(void) __attribute__((interrupt("WCH-Interrupt-fast")));
void EXTI0_IRQHandler(void) {
    if (EXTI_GetITStatus(EXTI_Line0) != RESET) {
        destroy(LONG_TASK_ID); // Zera a flag da tarefa longa na hora
        SHARED.sensor_port_A++;    // Registra o evento no IPC do usuário
        EXTI_ClearITPendingBit(EXTI_Line0);
    }
}

// ISR da Porta 2 (Sensor Crítico B)
void EXTI1_IRQHandler(void) __attribute__((interrupt("WCH-Interrupt-fast")));
void EXTI1_IRQHandler(void) {
    if (EXTI_GetITStatus(EXTI_Line1) != RESET) {
        destroy(LONG_TASK_ID); // Também aborta a tarefa se a segunda porta disparar
        SHARED.sensor_port_B++;    // Registra o evento no IPC do usuário
        EXTI_ClearITPendingBit(EXTI_Line1);
    }
}

// Exemplo de Tarefa Escada Escrita pelo Usuário
void task11(void) {
    for (u32 i = 0; i < 10000; i++) {
        ELOIDA_CHECKPOINT(LONG_TASK_ID); // Aborta se alguma interrupção limpou a flag
        
        // Execução do processamento longo do usuário...
        ExecutarAlgoritmo();
        
        ELOIDA_CHECKPOINT(LONG_TASK_ID);
    }
}

Por que essa arquitetura é eficiente?

  • Anti-Colisão Harmônica: O escalonador pulveriza os picos de processamento, impedindo o acúmulo indesejado de tarefas pesadas simultâneas.
  • Determinismo Estático: Não depende de ticks de relógio complexos para organizar as filas, economizando memória RAM preciosa em microcontroladores RISC-V.
  • Interrupção Amigável: Dá flexibilidade total ao desenvolvedor para usar o próprio hardware de interrupção da placa como uma ferramenta de controle de fluxo assíncrono.
